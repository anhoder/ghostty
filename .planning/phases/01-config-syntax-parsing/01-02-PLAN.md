  <action>
    The key design decision: how to differentiate conditional vs unconditional bindings for the same trigger in the HashMap.

    **Approach: Encode condition into the HashMap key.**

    The Set uses `HashMap = ArrayHashMapUnmanaged(Trigger, Value, Context(Trigger), true)`. Currently, two bindings with the same Trigger overwrite each other (last-write-wins). To support conditional + unconditional coexistence for the same trigger, we need the HashMap key to distinguish them.

    **Option A (recommended): Add `condition: ?Condition` to the Leaf, and use a separate storage path for conditional bindings.**

    Since CONF-04 says conditional bindings take priority over unconditional (evaluated at runtime in Phase 2), the cleanest approach for Phase 1 is:

    1. Add `condition: ?Condition = null` field to `Set.Leaf` struct:
    ```zig
    pub const Leaf = struct {
        action: Action,
        flags: Flags,
        condition: ?Condition = null,
    };
    ```

    2. Add `conditional_bindings: std.ArrayListUnmanaged(ConditionalEntry) = .{}` to `Set` struct, where:
    ```zig
    pub const ConditionalEntry = struct {
        trigger: Trigger,
        action: Action,
        flags: Flags,
        condition: Condition,
    };
    ```

    This keeps the existing `bindings` HashMap for unconditional bindings (zero change to existing lookup path) and adds a separate list for conditional bindings. Phase 2 will check `conditional_bindings` first (priority), then fall back to `bindings`.

    3. Modify `parseAndPutRecurse` to check `it.condition`:
       - If `it.condition != null`: store in `conditional_bindings` list instead of `bindings` HashMap
       - For last-write-wins: scan `conditional_bindings` for matching trigger+condition, replace if found, append if not
       - If `it.condition == null`: existing path unchanged

    4. Handle unbind for conditional bindings:
       - `[process=vim]ctrl+w=unbind` should remove the matching conditional entry
       - Plain `ctrl+w=unbind` only removes from unconditional `bindings` (existing behavior)

    5. Update `Set.deinit` to also clean up `conditional_bindings`.

    6. Update `Set.clone` (if it exists) to also clone `conditional_bindings`.

    7. Add `Condition.eql` method for comparing conditions:
    ```zig
    pub fn eql(a: Condition, b: Condition) bool {
        const tag_a = std.meta.activeTag(a);
        const tag_b = std.meta.activeTag(b);
        if (tag_a != tag_b) return false;
        return switch (a) {
            .process => |v| std.mem.eql(u8, v, b.process),
            .title => |v| std.mem.eql(u8, v, b.title),
            .var_ => |v| std.mem.eql(u8, v.name, b.var_.name) and std.mem.eql(u8, v.value, b.var_.value),
        };
    }
    ```

    8. Add `Trigger.eql` usage — the existing `Context(Trigger).eql` can be reused for trigger comparison in the conditional list scan.

    9. Add test block `test "set: parseAndPut conditional bindings"` at end of file with all behavior cases.

    IMPORTANT: The `conditional_bindings` ArrayList needs an allocator for management. Use the same `alloc` parameter passed to `parseAndPut`. Store items with string values duped via the allocator (same pattern as existing action string duplication in Set).

    IMPORTANT: Do NOT modify the existing `bindings` HashMap key type or Context. Unconditional bindings path must remain completely unchanged.
