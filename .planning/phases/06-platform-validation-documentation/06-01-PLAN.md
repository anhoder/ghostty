Insert a new `/// ## Conditional Bindings` doc comment section in `src/config/Config.zig` between the line `/// Key tables are available since Ghostty 1.3.0.` (line 1867) and the `keybind: Keybinds = .{},` field declaration (line 1868).

The section must follow the existing doc comment style (/// prefix, ## markdown headers, ```ini code fences for examples). Include:

1. **Overview**: Explain bracket syntax `[condition]trigger=action`, one condition per binding.

2. **Condition Types** (all three required):
   - `process=<name>` — foreground process name, supports glob (`*`, `?`). Example: `[process=vim]ctrl+w=close_surface` and `[process=nvim*]ctrl+w=close_surface`. Add note: process-name detection is polled ~every 200ms in the background; there is an eventual-consistency window of up to ~200ms. For latency-critical use cases, prefer `var=` conditions set via OSC 1337.
   - `title=<pattern>` — window title, supports glob. Example: `[title=vim:*]ctrl+s=write_scrollback_file:~/output.txt`.
   - `var=<name>:<value>` — user variable set via OSC 1337 SetUserVar, supports glob. Example: `[var=in_vim:1]ctrl+w=close_surface`. Note the colon separator between name and value. Briefly explain the OSC 1337 SetUserVar escape sequence: `\e]1337;SetUserVar=<name>=<base64-value>\a`.

3. **Priority**: Conditional bindings are checked first; if matched, they take priority over unconditional bindings for the same trigger. If no condition matches, the unconditional binding fires as fallback. Last-write-wins for same trigger+condition.

4. **Limitations** (brief subsection):
   - Only one condition per binding (no AND/OR).
   - Conditional bindings not supported for key sequences.
   - `global:` and `all:` prefixes not supported with conditional bindings.

After editing, run `zig ast-check src/config/Config.zig` to verify syntax.
